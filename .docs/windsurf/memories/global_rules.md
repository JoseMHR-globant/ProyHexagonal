## 2. Models by Layer

| Layer | Allowed Models | Restrictions |
|-------|----------------|--------------|
| **Controller** | `Dto` + Domain | — |
| **Usecase** | Domain only | Never Dto/Cdo/Entity |
| **Adapter** | `Cdo` + Entity | — |
| **Facade** | Domain + `Cdo` + Entity | — |

### 2.1 Model Requirements

```java
@Setter
@Getter
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class CustomerDto implements Serializable {
    private static final long serialVersionUID = 3456789089767543L; // Auto-generated by IDE
    // fields...
}
```

- ✅ Implement `Serializable` with auto-generated `serialVersionUID`
- ✅ Required annotations: `@Setter`, `@Getter`, `@Builder`, `@NoArgsConstructor`, `@AllArgsConstructor`

### 2.2 Modification Rules

To preserve backward compatibility with existing API consumers and Feign contracts.

| Model | Add fields | Modify fields | Remove fields |
|-------|------------|---------------|---------------|
| `Dto` | ✅ | ❌ | ❌ |
| `Cdo` | ✅ | ❌ | ❌ |

### 2.3 Minimum Effort Rule (Cdo)

`Cdo` models should only contain the fields strictly necessary for the requested functionality.

## 4. Controllers

### 4.1 Required Annotations

```java
@RestController
@RequestMapping("/v1/customers")
@Tag(name = "Customers")
@Validated
@RequiredArgsConstructor
public class CreateCustomerController {

    private final CreateCustomerUsecase usecase;
    private final CreateCustomerMapper mapper;

    @PostMapping
    public ResponseEntity<CustomerResponseDto> create(
            @Valid @RequestBody CustomerRequestDto request) {
        return ResponseEntity.ok(
            mapper.toDto(usecase.execute(mapper.toDomain(request))));
    }
}
```

### 4.2 Rules

- One endpoint per controller
- Always map: `Dto` → Domain → `Dto`
- Always annotate with `@Validated` for method-level `@Pattern` validation

### 4.3 Input Data Validators

All input data managed by the functionality **must be validated with a regex pattern** to prevent log injection, header injection, and other security vulnerabilities.

This applies to:
- **`@RequestBody`** fields (payload)
- **`@RequestHeader`** values
- **`@RequestParam`** values
- **`@PathVariable`** values

Use the standard Jakarta `@Pattern` annotation (`jakarta.validation.constraints.Pattern`):

#### Usage examples

```java
// In Dto fields
public class CustomerRequestDto implements Serializable {
    @Pattern(regexp = "^[a-zA-Z0-9\\-_.@\\s]{0,255}$")
    private String name;

    @Pattern(regexp = "^[A-Z]{2}$")
    private String countryCode;
}

// In controller method parameters
@GetMapping("/{id}")
public ResponseEntity<CustomerResponseDto> get(
    @PathVariable
    @Pattern(regexp = "^[0-9a-f\\-]{36}$") String id,
    @RequestHeader("X-Correlation-Id")
    @Pattern(regexp = "^[a-zA-Z0-9\\-]{0,64}$")
        String correlationId,
    @RequestParam(required = false)
    @Pattern(regexp = "^[a-zA-Z0-9\\-_.\\s]{0,255}$")
        String filter) {
    // ...
}
```

#### Rules

- **Every** string input field in Dto models must have a `@Pattern` annotation with a restrictive regex
- Use a regex **tailored to the expected format** (e.g., UUID, country code, email)
- **Reject** inputs containing: newlines (`\n`, `\r`), tabs, null bytes, CRLF sequences, and any characters that could be used for log injection or header manipulation
- Numeric fields (`Integer`, `Long`, `BigDecimal`) do not need regex validation as they are type-safe
- For `@RequestHeader` and `@RequestParam` used directly in the controller method signature, apply `@Pattern` at the parameter level
- For nested Dto fields, add `@Valid` to propagate validation

## 5. Mappers

| Layer | Source | Target |
|-------|--------|--------|
| **Controller** | `Dto` ↔ Domain | Bidirectional |
| **Usecase** | Domain ↔ Domain | Only between domain models |
| **Facade** | Domain ↔ `Cdo`/Entity | Bidirectional |

### 5.1 MapStruct convention — Interface + `Gs` implementation

Every mapper has a **generic parent interface** (declares the contract with type parameters bound by domain/DTO superclasses) and a **`Gs` concrete interface** that binds those generics for the DCP bank.

```java
@Mapper(
    componentModel = MappingConstants.ComponentModel.SPRING,
    unmappedTargetPolicy = ReportingPolicy.IGNORE)
@TargetMultiBean(bankId = BankIdConstant.DCP)
public interface GsXxxMapper extends XxxMapper<ConcreteSource, ConcreteTarget> {}
```

### 5.2 Do NOT redeclare methods that don't add anything

If the `Gs` interface only binds generics and adds **no** custom `@Mapping` or logic, the body must be empty `{}`. MapStruct generates the implementation automatically from the bounds in `extends`. Adding a redundant `@Override` of the parent method (without `@Mapping`) is forbidden — it adds noise and triggers reviewer comments.

## 6. General Rules

### 6.1 Forbidden ❌

#### Exceptions

- `e.printStackTrace()` → use logger
- Swallow exceptions (empty catch without log or rethrow)
- `throw new RuntimeException(e)` without context

#### Code

- Magic numbers/strings without constants
- Methods with more than 5 parameters → use objects
- Cascading `instanceof` → use polymorphism
- Deeply nested `if-else` (> 4 levels)
- Static variables in beans (`@Component`, `@Service`, `@Controller`, `@Repository`)
- `Dto`, `Cdo` or Entity in usecases or common services
- Returning `null` → use `Optional`
- Non-static inner classes
- Entry/exit logs in controllers

#### Collections and Nulls

- Returning `null` instead of empty collection → use `Collections.emptyList()`
- `list.size() == 0` → use `list.isEmpty()`
- Modifying collections while iterating

#### Strings

- Concatenation with `+` in loops → use `StringBuilder`
- `string == "value"` → use `"value".equals(string)`

#### Spring

- `@Autowired` on fields → use constructor injection
- Business logic in controllers
- `@Transactional` on private methods (does not work)

#### Logging

- Logging sensitive data (PII, passwords, tokens, card numbers)
- `log.info()` for errors or exceptions
- Logs inside loops (generates thousands of lines)
- `log.debug()` with expensive string concatenation without checking level
- `log.error()` without proper justification → only use for alertable errors

#### Redis Restrictions

- `KEYS` command with wildcards (`KEYS user:*`) → blocks Redis, use `SCAN` if necessary
- `@CacheEvict(allEntries = true)` → avoid mass eviction, use specific keys
- Wildcard patterns in cache keys → design keys to avoid pattern searches
- Prefer TTL expiration over manual eviction

#### SQL Restrictions

- `SELECT *` → always specify required columns explicitly
- Queries without `WHERE` clause on large tables
- `LIKE '%value%'` → leading wildcard prevents index usage, use `LIKE 'value%'`
- Missing indexes on frequently queried columns
- N+1 query patterns → use `JOIN FETCH` or batch queries

#### Flyway

> 📖 Full Flyway standards: see Skill [`flyway.md`](%USERPROFILE%\.codeium\windsurf\skills\flyway.md)

### 6.2 Required ✅

#### Code

- Constants: `private static final`
- Injection: `@RequiredArgsConstructor` + `private final`
- Exceptions: `<base-package>.error.commons.exception`

#### Logging

- Use correct levels: `ERROR` (alertable failures), `WARN` (recoverable), `INFO` (important events), `DEBUG` (development)
- `log.error()` must be justified → only for errors that require alerting
- Include context: transaction IDs, user IDs, operation type
- Log exceptions with `log.error("message", exception)` (includes stacktrace)
- Use placeholders `{}` instead of concatenation: `log.info("User {} created", userId)`

## 7. Code Formatting

### 7.1 Google Java Style Guide

All Java code must follow the [Google Java Style Guide](https://google.github.io/styleguide/javaguide.html).

### 7.2 Key Formatting Rules

| Rule | Value |
|------|-------|
| Indentation | 2 spaces (no tabs) |
| Max line length | 80 characters |
| Braces | K&R style (opening brace on same line) |
| Imports | No wildcard imports (`*`) |
| Import order | Static imports first, then regular imports, alphabetically |

### 7.3 Rules

- ✅ Format code before every commit
- ✅ Configure IDE to format on save
- ✅ **Only format new or modified lines** (do not reformat entire files)
- ❌ Do NOT commit unformatted code
- ❌ Do NOT mix formatting changes with functional changes in the same commit
- ❌ Do NOT reformat existing code unless explicitly requested

### 7.4 Static Analysis (PMD)

All code is validated against PMD rules during the build (`mvn pmd:check`). Violations will fail the build.

> 📖 Full PMD ruleset reference: see Skill [`pmd-ruleset.md`](%USERPROFILE%\.codeium\windsurf\skills\pmd\SKILL.md)

## 8. Exceptions

### 8.1 Platform Exceptions

Feign clients configured with platform settings throw the following runtime exceptions from library `mb-api-error-commons` (package `<base-package>.error.commons.exception`):

| Exception | HTTP Code | Description |
|-----------|-----------|-------------|
| `BadRequestException` | 400 | Invalid request parameters |
| `UnauthorizedException` | 401 | Authentication required |
| `ForbiddenException` | 403 | Access denied |
| `NotFoundException` | 404 | Resource not found |
| `BusinessValidationException` | 412 | Business validation failed |
| `BusinessUnprocessableEntityException` | 422 | Unprocessable business entity |
| `HttpException` | Other | All other HTTP error codes |

### 8.2 Rules

- ✅ Whenever possible, use exceptions from `<base-package>.error.commons.exception`
- ✅ Let platform exceptions propagate (handled by global error handler)
- ✅ Log context before rethrowing if needed
- ❌ Do NOT catch and swallow platform exceptions
- ❌ Do NOT wrap platform exceptions in `RuntimeException`
- ❌ Do NOT create custom exceptions for HTTP error scenarios

### 8.3 Examples

```java
// Validate that the orderId belongs to the customer's orders
if (!customerOrders.contains(orderId)) {
    throw new ForbiddenException(
        "ERROR_CODE", "MESSAGE");
}
```

## 9. Package Structure

### 9.1 Base Package

```
com.example.{service-name}
```

### 9.2 Common Package (`common/`)
 
| Subpackage | Purpose | Contains |
|------------|---------|----------|
| `config/` | Spring configuration | `@Configuration`, `@ConfigurationProperties` |
| `adapter/client/{client}/` | Feign clients | `{Client}Client.java` + `model/{Model}Cdo.java` |
| `adapter/repository/` | JPA repositories | `{Entity}Repository.java` |
| `domain/{resource}/` | Domain models | `{Model}.java` |
| `facade/{resource}/` | Facades | `{Resource}Facade.java` + `mapper/{Resource}FacadeMapper.java` |
| `service/` | Shared services | Used by multiple usecases |
| `util/` | Utilities | Common helpers |
| `monitor/` | Cross-cutting | Aspects, filters, interceptors |
| `validator/` | Validation | Custom Jakarta validations |

### 9.3 Feature Package (`{feature}/`)
 
| Subpackage | Purpose | Contains |
|------------|---------|----------|
| `controller/` | REST entry point | `{Feature}Controller.java` |
| `controller/model/` | Request/Response DTOs | `{Feature}RequestDto.java`, `{Feature}ResponseDto.java` |
| `controller/mapper/` | DTO ↔ Domain mappers | `{Feature}Mapper.java` |
| `controller/validation/` | Custom validators | Jakarta constraint annotations |
| `queue/` | JMS entry point | Alternative to controller |
| `scheduler/` | Batch entry point | Alternative to controller |
| `usecase/` | Business logic | `{Feature}Usecase.java` |

## 10. Data Flow

### 10.1 Core Principles

| Principle | Description |
|-----------|-------------|
| **Usecase Isolation** | Usecase is "pure" - only knows Domain models |
| **Mappers at Boundaries** | Mappers only exist in Controller and Facade layers |
| **No Layer Skipping** | Controller never calls Adapter directly |
| **Inward Dependency** | Outer layers depend on inner layers, never the reverse |

### 10.2 Layer Flow Summary
 
| Layer | Input | Mapper | Output | Calls |
|-------|-------|--------|--------|-------|
| **Controller** | `Dto` | Controller Mapper | `Domain` | Usecase |
| **Usecase** | `Domain` | — | `Domain` | Facade, Service |
| **Facade** | `Domain` | Facade Mapper | `Cdo`/`Entity` | Client, Repository |
| **Adapter** | `Cdo`/`Entity` | — | `Cdo`/`Entity` | External systems |
